# 分域部署方案（Next C 端 + PHP 后台 + 反向代理门禁）

本文档用于后续 Agent 实施，目标是将 C 端与后台分域部署，并把“入口可见性”门禁从 `php-server` 应用内移到反向代理层。

---

## 1. 目标与边界

### 1.1 目标

在同一台 Linux 服务器（CentOS 或 Ubuntu 24）上：

1. Docker 运行 `mysql:5.6`。
2. `php-server`（后台 Admin）连接该 MySQL。
3. `dinngoo-site`（Next.js）连接同一 MySQL。
4. 反向代理（推荐 Caddy，可选 Nginx）做域名分流：
   - `www.<主域名>` -> Next.js C 端
   - `op.<主域名>` -> PHP Admin
5. `op` 域名通过“入口 token -> cookie 30 天 -> 自动续期 -> 去参跳转”实现可见性门禁。

### 1.2 边界

- **不在 php-server 应用内再做 entry_token 校验**（已移除相关代码）。
- 门禁逻辑统一放在反向代理层，便于后续替换部署方式或统一审计。

---

## 2. 推荐选型

### 2.1 推荐：Caddy

理由：

- 自动 HTTPS 与证书续期，配置更短。
- 当前需求是“分域 + 简单门禁 + 续期”，无需引入复杂脚本层。
- 便于后续 Agent 低风险落地。

### 2.2 备选：Nginx

适用场景：

- 团队已有成熟 Nginx 运维体系。
- 未来门禁策略需要更复杂逻辑（多组 token、细粒度策略、扩展脚本）。

---

## 3. 服务拓扑（建议）

```text
Internet
   |
   v
[ Caddy (Host Native Install) ]
   |-- www.example.com --> nextjs:3000 (Docker)
   `-- op.example.com  --> php-nginx:80 (Docker)

Docker Network (internal)
   |- mysql56:3306
   |- nextjs:3000
   `- php-server (php-fpm + nginx)
```

说明：

- 仅主机上的 Caddy 对外暴露 80/443。
- MySQL、PHP-FPM、Next.js 保持容器网络隔离，不对公网暴露。

---

## 4. 门禁规则（op 域名）

### 4.1 规则定义

以 `entry_token` 为 query 参数名（可配置）：

1. 请求 `op` 域名时，若 URL 上带 `entry_token` 且值合法：
   - 下发 HttpOnly Cookie（有效期 30 天）。
   - 立即 302/303 重定向到去掉 query 的干净 URL。
2. 若请求无 query token，但 Cookie 合法：
   - 放行到 PHP Admin。
   - 刷新 Cookie 过期时间（滑动续期）。
3. 两者都不合法：
   - 返回 404（隐藏后台入口）。

### 4.2 安全建议

- Cookie：`HttpOnly`、`Secure`、`SameSite=Lax`。
- 仅在 `op` 域名下设置，不与 `www` 共用。
- token 不写仓库；使用服务器环境变量或独立 secrets 文件。
- 轮换 **entry_token** 或 cookie 中保存的 token 后，**旧值立即失效**（环境变量仅保留当前 `OP_GATE_TOKEN` / `MAINT_GATE_TOKEN`，无 `*_OLD` 并行窗口）。

---

## 5. 数据与连接约定

### 5.1 MySQL

- 版本：`5.6`（按当前约束）。
- 仅内网可达，不对公网开放 3306。
- 至少创建两个数据库账号（建议）：
  - `app_php`：给 php-server
  - `app_next`：给 Next.js
- 权限最小化到指定库与必要操作。

### 5.2 php-server

- Admin 入口使用 `op.<主域名>/admin/...`。
- 应用内不再依赖 `entry_token` 配置（相关文件已移除）。

### 5.3 Next.js

- C 端使用 `www.<主域名>`。
- 数据连接配置通过 `.env`（容器注入），不要提交真实口令。

---

## 6. Agent 实施顺序（建议）

1. **服务器准备**
   - 安装 Docker / Docker Compose。
   - 开放 80/443（SSH 端口按运维策略）。
2. **容器网络与服务编排**
   - 使用两个项目各自的 compose：`php-server/docker-compose.yml`、`dinngoo-site/docker-compose.yml`（如无则新增）。
   - 编排 `mysql56`、`php-server`、`nextjs` 三类容器。
   - 不在 Docker 内跑 Caddy。
3. **数据库初始化**
   - 建库、建用户、导入基础数据。
4. **应用配置**
   - php-server 与 nextjs 分别注入 DB 连接。
5. **域名与证书**
   - 配置 `www`、`op` A 记录。
   - 主机安装 Caddy，配置自动签发证书并反代到容器服务。
6. **配置 op 门禁**
   - query token 发卡、cookie 续期、非法 404。
7. **联调验证**
   - `www` 正常访问 C 端。
   - `op` 无 token 404；带 token 可进入并写 cookie；后续仅 cookie 放行。
8. **上线后运维**
   - token 轮换流程（双 token 窗口）。
   - 访问日志审计与失败告警。

## 6.1 两个项目各自 Compose（推荐落地）

按你的部署偏好：

- **`php-server/docker-compose.prod.yml`**
  - 启动 **MySQL 5.7**（镜像与仓库 `docker-compose.yml` 一致）+ `php` + `nginx`。
  - 对主机开放 `127.0.0.1:18080`（仅供 Caddy 反代）；数据库口令见仓库根目录 `.env`（示例：`deploy/env.compose.prod.example`）。
- **`dinngoo-site/docker-compose.prod.yml`**
  - 启动 **Next.js**：**薄镜像** + **只读挂载** **`./.next/standalone-runtime`**（由 **`./deploy/stage-standalone.sh`** 生成：standalone + `public` + `.next/static`）。
  - 日常更新：宿主 **`stage-standalone`** 后 **`docker compose ... restart next`**，**不必**重建镜像；仅 **Dockerfile** 变更时 **`up -d --build`**。
  - 对主机开放 `127.0.0.1:13000`（仅供 Caddy 反代）；环境变量见 `.env.production.local`（示例：`deploy/env.production.docker.example`）。

**容器日志（两文件一致约定）**：默认 **`logging.driver: json-file`**（`max-size` / `max-file`）。按日筛选与错误关键词粗分析可用 **`supports/server-maintain/scripts/analyze-docker-logs.mjs`**（配置 `DOCKER_LOG_CONTAINERS` 等）。若你**自行**改为 **syslog** 驱动，需在宿主机配置 **rsyslog**（或同类）接收 `unixgram:///dev/log`，无单独仓库模板。

首次准备：

```bash
docker network create dinngoo_net || true
```

启动顺序建议：

```bash
# 1) php-server（含 mysql）
cd /srv/dinngoo-room/php-server
cp deploy/env.compose.prod.example .env   # 再编辑 .env 填写强口令
docker compose -f docker-compose.prod.yml up -d mysql
docker compose -f docker-compose.prod.yml up -d --build php nginx

# 2) dinngoo-site（宿主机 standalone 汇总后只读挂载；需 Node 22）
cd /srv/dinngoo-room/dinngoo-site
cp deploy/env.production.docker.example .env.production.local   # 再编辑
./deploy/stage-standalone.sh
docker compose -f docker-compose.prod.yml up -d --build
# 仅业务更新：git pull && ./deploy/stage-standalone.sh && docker compose -f docker-compose.prod.yml restart next
```

## 7. 验收清单

- [ ] `www.<主域名>` 正常打开 Next.js 首页。
- [ ] `op.<主域名>/admin/login/index.html` 未带 token 返回 404。
- [ ] 带 `?entry_token=...` 首次访问可进入，并自动跳转到无 query URL。
- [ ] 浏览器存在 `op` 域 cookie，刷新后仍可访问。
- [ ] 30 天有效期可滑动续期（查看响应头/开发者工具验证）。
- [ ] Caddy 能正确反代：`www -> 127.0.0.1:13000`，`op -> 127.0.0.1:18080`。
- [ ] php-server 与 nextjs 均可访问 MySQL，核心页面无连接错误。
- [ ] 主机 Caddy 重启后可自动恢复路由与证书。

---

## 8. 实施注意

- CentOS 若为旧版本，需先确认系统与依赖支持周期；条件允许优先 Ubuntu 24。
- MySQL 5.6 已较老，后续应单独规划升级窗口（5.7/8.0）。
- 切勿把 token、数据库口令、私钥写入仓库或部署脚本明文。

## 9. 可直接复用的模板文件（Caddy 分域 + 运维站）

本目录下已提供可复制修改的模板：

- [**templates/Caddyfile.split-domains.example**](./templates/Caddyfile.split-domains.example)
  - `www`/`op`/`maint` 分流；`op`/`maint` 为 `entry_token` 门禁（发 cookie、续期、404 隐藏）；`maint` 为运维静态站；访问日志 JSON + 滚动参数见文件内注释。

约定说明：

- Compose 以两个项目各自维护为准（`php-server` 与 `dinngoo-site`）。
- Caddy 使用主机原生安装，不作为容器运行。

使用建议：

1. 复制 `Caddyfile` 模板到服务器 Caddy 配置目录（例如 `/etc/caddy/Caddyfile`）。
2. 替换域名、上游地址、token 环境变量。
3. 启动前先验证 Caddy 日志目录可写（如 `/var/log/caddy`）。
4. 用 `curl` 验证 `op` 门禁三段逻辑（无 token 404、query 发卡、cookie 放行）。

## 10. 运维报表、指标与备份（`supports/server-maintain`）

- **域名**：当前测试环境为 **`maint.dinngoo.xyz`**（见 `templates/Caddyfile.split-domains.example`）；其它环境可将 Caddy `server` 名改为 `maint.<主域名>`。`file_server` 根目录指向 **`MAINT_STATIC_ROOT`**（与工具集里的 `REPORT_DIR` 一致）。
- **门禁**：与 `op` 相同模式（`entry_token` Query + HttpOnly Cookie + 滑动续期），使用独立变量 **`MAINT_GATE_TOKEN`** 与 cookie 名 **`maint_gate`**。**`OP_GATE_TOKEN` 与 `MAINT_GATE_TOKEN` 必须为不同随机串**，勿与后台入口共用同一 token。
- **工具**：详见 [**server-maintain/README.md**](../server-maintain/README.md) — 日报 HTML、`metrics.json`、MySQL 全量备份、**手动**执行 `rotate-gate-tokens.mjs` 轮换 token（不要为轮换配置 cron）。
- **首次部署**：将 `server-maintain/public/dashboard.html` 复制到 `MAINT_STATIC_ROOT`，与 `collect-metrics.mjs` 生成的 `metrics.json` 同目录。
