# 安全事件排查总结（2026-05-02）

> **说明**：正文中的 **PM2**、**Next 15.x** 等为当时运行形态与版本。当前 C 端为 **`dinngoo-site` Docker**：**standalone** 汇总目录 **`.next/standalone-runtime`** 只读挂载，日常流程见 [测试环境部署和更新维护.md](./测试环境部署和更新维护.md)。下文保留事件原貌备查。

## 1. 背景

- 目标主机：阿里云 ECS（公网可访问测试机）
- 事件类型：云安全中心告警「Web 应用创建异常子进程」
- 关键告警信息（摘录）：
  - 父进程：`next-server (v15.2.4)`（`/usr/bin/node`）
  - 子进程：`/usr/bin/dash`，命令行为 `/bin/sh -c whoami`
  - 触发时间：`2026-05-02 00:17:28`
- 当时服务器状态：未正式上线，处于备案期，公网可访问。

## 2. 排查目标

1. 判断是否为误报（正常业务行为）或异常执行（潜在 RCE）。
2. 判断是否存在持久化后门/横向移动痕迹。
3. 在不影响当前联调的前提下，完成应急处置与版本修复。

## 3. 关键证据与发现

## 3.1 应用侧证据（Next.js/PM2）

- 在 `dinngoo-site` 的 PM2 错误日志中出现异常内容（非正常业务日志）：
  - `NEXT_REDIRECT ... 'VULN_633940\n'`
  - `NEXT_REDIRECT ... 'ecs-user\n'`
  - `Command failed: ps aux | grep xmrig | grep -v grep`
- 上述内容与云告警（`/bin/sh -c whoami`）相互印证，说明曾发生过异常命令执行链路。

## 3.2 系统侧证据（主机与登录）

- 未发现以下持续化迹象：
  - 可疑常驻进程（如矿工、反弹 shell、异常守护）
  - root/ecs-user 异常 crontab
  - `/tmp`、`/var/tmp`、`/home/ecs-user` 下明显恶意落地脚本（排查窗口内）
- SSH/Sudo 记录主要对应已知运维来源与已执行运维命令，未见明显陌生 root 持续操控迹象。

## 3.3 漏洞面判断

- 事件发生时 Next.js 为 `15.2.4`，属于已披露高危漏洞受影响版本区间。
- 后续已升级并重建为 `Next.js 15.5.15`（修复线），并重新部署运行。

## 4. 已执行处置

1. 升级 `dinngoo-site` 依赖：
   - `next` 升至 `15.5.15`

## 5. 结论

- 本次事件**不建议定义为纯误报**。
- 更合理判断是：在旧版 Next.js 暴露期内，出现了疑似漏洞探测/利用行为，且有短时命令执行迹象。
- 暂未发现明确持久化后门或持续控制证据，但不能据此完全排除历史瞬时入侵成功。
