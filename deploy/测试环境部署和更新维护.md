# 测试环境部署与更新维护说明

本文档描述**当前**测试机（Ubuntu + Docker + 宿主机 Caddy）的目录、端口、日常运维方式与数据库变更约定。**不包含**任何账号口令、GitHub Token、反代门禁 Token 等敏感内容；凭据仅存服务器本地或密码管理器。

- **分域架构、C 端 Next + 后台 PHP、反代层门禁**：见同目录 [**分域部署方案.md**](./分域部署方案.md)。  
- **Caddy 配置模板**：[**templates/Caddyfile.split-domains.example**](./templates/Caddyfile.split-domains.example)。

## 2. 拓扑与端口

仅 **Caddy** 对公网暴露 **80/443**。容器栈只绑定 **127.0.0.1**：

| 监听 | 服务 |
|------|------|
| **127.0.0.1:3306** | MySQL（`php-server` 的 `docker-compose.prod.yml` 中 `mysql`） |
| **127.0.0.1:18080** | PHP 站点 Nginx（ThinkPHP `public`） |
| **127.0.0.1:13000** | Next.js 生产进程 |

Caddy 将 `www.*` → `13000`，`op.*` → `18080`（并做 query/cookie 门禁），详见 [**分域部署方案.md**](./分域部署方案.md)。

**防火墙 / 安全组**：放行 **22**（或你的 SSH 端口）、**80**、**443**；**不要**对公网放行 **3306**。

## 3. 部署目录与密钥文件

| 路径 | 说明 |
|------|------|
| **`/srv/dinggu-room/php-server`** | 后台 PHP 仓库；`docker compose -f docker-compose.prod.yml` 在此执行 |
| **`/srv/dinggu-room/dinngoo-site`** | C 端 Next 仓库；宿主 **`stage-standalone`** 生成 **`.next/standalone-runtime`**，Compose **只读挂载**该目录 |
| **`/srv/dinggu-room/php-server/.env`** | Compose 变量：`MYSQL_ROOT_PASSWORD`、`MYSQL_USER`、`MYSQL_PASSWORD` 等（**勿提交**；示例见 `php-server/deploy/env.compose.prod.example`） |
| **`/srv/dinggu-room/dinngoo-site/.env.production.local`** | Next 运行时环境变量（**勿提交**；示例见 `dinngoo-site/deploy/env.production.docker.example`）；**`DATABASE_*` 须与 MySQL 中 `room` 用户一致** |
| **`/srv/dinggu-room/.secrets/op_gate_token`** | 反代 `op` 门禁用 token（由 Caddy `EnvironmentFile` 读入 `OP_GATE_TOKEN`）；权限建议 `600` |
| **`php-server/application/config_local/database.local.php`** | 容器内 PHP 连库（`hostname` 多为 `mysql`）；**勿提交** |

外部 Docker 网络（两栈共用）：**`dinggu_net`**（`docker network create dinggu_net`）。

本地开发仍可用 **`php-server`** 仓库根目录 **`docker-compose.yml`**（8080 / 默认网络）；与 **`docker-compose.prod.yml`** 区别见该文件头注释。

## 4. Docker 常用命令（部署/重启）

在 **`php-server`** 目录（需能访问 Docker，例如 `sudo docker` 或用户位于 `docker` 组）：

```bash
cd /srv/dinggu-room/php-server
docker compose -f docker-compose.prod.yml up -d mysql
# 健康后
docker compose -f docker-compose.prod.yml up -d --build php nginx
```

在 **`dinngoo-site`** 目录（先确保 MySQL 已起且网络 `dinggu_net` 已存在；**宿主机需安装 Node 22**）：

```bash
cd /srv/dinggu-room/dinngoo-site
./deploy/stage-standalone.sh   # 或 ./deploy/build.sh：npm ci + build + 汇总 standalone/public/static
docker compose -f docker-compose.prod.yml up -d --build   # 首次 / Dockerfile 变更
# 日常仅代码更新：./deploy/stage-standalone.sh && docker compose -f docker-compose.prod.yml restart next
```

若容器读挂载目录报 **EACCES**：镜像内用户 **uid 1000**，可 `sudo chown -R 1000:1000 .next/standalone-runtime`。

**PHP 与 Nginx**：单独重建 **`php`** 容器后，建议 **`docker compose ... restart nginx`**，避免 Nginx 仍解析到旧 PHP 容器 IP（同类问题见运维笔记）。

查看状态：`docker compose -f docker-compose.prod.yml ps`、`logs`。

**注意**：`gunzip | docker compose exec -T mysql mysql ...` 与 `echo 密码 | sudo -S` 不能同管道，否则 stdin 冲突；导入 SQL 请在 **`sudo bash`** 或脚本内执行 `docker compose`，见运维笔记。

## 5. Git / GitHub CLI（`gh`）

- 推荐使用 **`gh auth login`**（HTTPS + PAT）并 **`gh auth setup-git`**；远程 **不要** 在 URL 里写死 Token。  
- 私有库需 PAT 具备对应 **repository** 权限，或账号 **SSH 公钥** 已添加到 GitHub。  
- 更新代码：`cd` 至对应仓库后 `git fetch` / `git pull --ff-only`；**php-server** 多为 `compose restart`；**dinngoo-site**：**`./deploy/stage-standalone.sh`**（或 **`./deploy/build.sh`**）后 **`docker compose -f docker-compose.prod.yml restart next`**（镜像/Dockerfile 未改时无需 `--build`）。

## 6. Caddy 与 HTTPS

- 主配置：**`/etc/caddy/Caddyfile`**  
- 门禁变量：**`/etc/caddy/env.deploy`**（由 `systemd` `EnvironmentFile` 加载）  
- 日志目录：**`/var/log/caddy`**（属主 `caddy`）  
- **Let’s Encrypt** 需域名 **A 记录** 指向本机；未解析时证书会失败。  
- Ubuntu 自带 Caddy 的 Caddyfile **不要** 使用已废弃的 `auto_https on` 写法；模板已按发行版兼容方式维护。

## 7. 数据库与导入约定

- **逻辑库名**：`room_controll`（与迁移、示例一致）。  
- **应用用户**：多为 `room`；**具体口令以 `.env` / `database.local.php` 为准**，本文不记录。  
- **字符集**：**utf8mb4**。  
- **导入顺序（新库）**：建库与用户（Compose 或手工）→ 导入 **`.sql` / `.sql.gz`** → 再执行 **`php-server/migrations/dinggu-mobile-1.0/*.up.sql`**（升序）。  
- **dump 示例文件名**：`php-server/data/room_controll_*_mysql_data.sql.gz`（按仓库实际为准）。  
- 大批量导入前可在 MySQL 内调大 **`max_allowed_packet`**。

## 9. 数据库升级与回滚策略

约定细节以 **`php-server/migrations/README.md`** 与 **`php-server/migrations/dinggu-mobile-1.0/README.md`** 为准。以下为**运维规范**（不记录某次已执行过的具体历史）。

### 9.1 升级前

1. **全库备份**（`mysqldump`、快照或云备份），确认可恢复到升级前。  
2. 确认**目标代码版本**与迁移脚本对应关系。  
3. 阅读批次 `README.md` 中的依赖说明；跨脚本依赖须按文件名**三位序号升序**执行 UP。

### 9.2 执行升级（UP）

- **命名**：`NNN-主题.up.sql`；**顺序**：`php-server/migrations/dinggu-mobile-1.0/*.up.sql` **升序**。  
- **幂等性**：多数 UP **非幂等**；重复执行可能因「列/表已存在」失败。失败后应从**失败文件**重试，避免重复已成功脚本。

**主机直跑 `mysql` 的示例**（口令勿写入文档）：

```bash
cd /srv/dinggu-room/php-server
for f in migrations/dinggu-mobile-1.0/*.up.sql; do
  echo "==> $f"
  mysql -h127.0.0.1 -uroot -p*** room_controll < "$f"
done
```

**Docker 内执行单条 UP**（避免 `sudo -S` 与管道争用 stdin，在 `sudo bash` 中执行）：

```bash
cd /srv/dinggu-room/php-server
sudo bash -c 'ROOTPW=$(grep ^MYSQL_ROOT_PASSWORD= .env | sed "s/^MYSQL_ROOT_PASSWORD=//"); \
  docker compose -f docker-compose.prod.yml exec -T mysql mysql -uroot -p"$ROOTPW" room_controll < migrations/dinggu-mobile-1.0/NNN-xxx.up.sql'
```

### 9.3 回滚（DOWN）

- **命名**：`NNN-主题.down.sql`；**顺序**：与 UP **相反**（序号从大到小）。  
- **风险**：DOWN 常含删表/删列，执行前须**备份**。

```bash
# 文件名仅为示意
mysql -h127.0.0.1 -uroot -p*** room_controll < migrations/dinggu-mobile-1.0/091-....down.sql
```

### 9.4 导入业务 dump 与迁移的关系

- **典型顺序**：建库与用户 → 导入 **`php-server/data/room_controll_*_mysql_data.sql(.gz)`** → 再执行所需 **`*.up.sql`**。  
- 若 dump 已含部分结构，须与迁移 README 对齐，避免重复执行冲突。  
- 大文件注意 **`max_allowed_packet`** 与超时；**`.gz`** 在容器外可用 `gunzip -c ... | docker compose exec -T mysql ...`（须在 **root  bash** 内整条管道，勿与 `sudo -S` 从 stdin 读密码混用）。

### 9.5 失败处理

- 某条 UP 失败后：**停止**后续脚本，保留日志，修复后从**该文件**重试。  
- 需整体撤销时：**有备份则优先恢复备份**；或在确认 DOWN 范围足够时再执行 DOWN。

## 10. 仅代码更新（无库变更）

1. `git pull` 更新 **`php-server`** / **`dinngoo-site`**。  
2. **php-server**：无 Docker 层变更时多仅需 **`docker compose -f docker-compose.prod.yml restart php nginx`**，必要时清 **`runtime`** 缓存；依赖变更则在容器内或 CI 执行 **`composer install`**。  
3. **dinngoo-site**：**`./deploy/stage-standalone.sh`**（或 **`./deploy/build.sh`**）后 **`docker compose -f docker-compose.prod.yml restart next`**；Dockerfile 变更时再 **`up -d --build`**。  
4. ThinkPHP **`runtime` / `public/upload`**：挂载卷时注意权限与清理缓存；Next 静态资源以构建产物为准。

## 12. 安全与合规

- **PAT、sudo 密码、门禁 token** 出现在聊天或日志中即应视为泄露，**轮换**并收紧 **Fine-grained PAT** 仓库范围。  
- **勿将** `.env`、`.secrets`、`database.local.php`、**`*.local.php`** 提交到 Git。  
- 对外服务默认 **HTTPS**；按需增加 IP 限制、VPN 等。
